# 데이터 모델의 이해

> 이 내용은 2026 이기적 SQLD 개발자 기본서를 참고하였습니다.


### 정의

**데이터 모델** : 현실의 대상을 데이터로 저장하고 활용하기 위한 설계도

**데이터 모델링** : 데이터 모델을 설계하는 과정


### 데이터 모델링의 특징

1. 단순화
- 약속된 표기법 이용. 현실 세계의 데이터는 복잡하지만, DB에 저장할 때는 단순하게.

2. 추상화
- 구체적인 데이터를 모두 표현하고자 하면 데이터가 복잡해진다. 특징만을 추려 추상적으로 표현한다.
- ex) 사과, 바나나, 멜론, 배 ,... -> 과일

3. 명확화(=해석 용이성)
- 위 두 방법으로 데이터를 이해하기 쉽고 명확히 파악가능한 형태로 표현하는 것.


### 데이터 모델링 관점

1. **데이터** 관점 모델링
- 업무가 어떤(What) 데이터와 관련있는지 모델링
- ex) '회원가입' 업무에는 '이름', '회원ID', '비밀번호', '나이' 등 데이터가 필요.

2. **프로세스** 관점 모델링
- 업무가 어떻게(How) 진행되는지 관점에서 모델링.
- 어떤 데이터가 필요한지보다 어떤 흐름으로 업무가 진행되는지에 중점.

3. **상관** 관점 모델링
- 업무(How)에 따라 데이터(What)와 어떻게 작용하는지 관점에서 모델링.
- 이런 관계를 CRUD매트릭스 형태로 분석해 표현.

*CRUD 매트릭스 : 생성Create, 읽기Read, 수정Update, 삭제Delete의 4가지 프로세스.*


### 데이터 모델링 중요성

1. 파급효과
- 설계 없이 개발하면 요구사항 변경이나 기능 추가시 예상치 못한 파급효과가 발생할 수 있다.

2. 간결한 표현
- 정보 요구 사항을 글로만 표현하면 이해가 어렵거나 사람마다 해석이 다를 수 있다. 특히 여러 직군의 협업을 요하는 경우 해석 차이로 인한 소통 오류를 겪을 수 있다. 따라서 그림으로 시각화하여 공통된 해석을 용이하게 얻도록 하는데, 데이터 모델링이 이런 시각화를 가능하게 하는 설계도를 제공.

3. 데이터 품질
- 데이터 모델링을 제대로 하지 않으면 데이터의 품질이 떨어진다. 즉, **데이터를 신뢰할 수 없다**.
- (1) 중복 : 동일 데이터를 여러 저장소에서 관리하면 데이터가 중복되어 데이터 품질이 하락함.
- (2) 비유연성 : 데이터와 프로세스 설계가 너무 긴밀하게 연관되면 업무 변화나 새로운 요청에 쉽게 대응하지 못할 수 있다. 유연한 설계를 해야 함.
- (3) 비일관성 : 같은 상황인데도 데이터가 다르게 저장되면(=입력 규칙/흐름이 일관되지 못하면) 데이터의 신뢰성 떨어짐. 업무 흐름이 항시 동일하게 적용되어야.


### 좋은 데이터 모델링의 기준

1. 현업 요구사항과 비즈니스 규칙 반영
2. 데이터 중복 방지 및 재사용 기능
3. 업무 관련자 간 의사소통에 용이
4. 데이터의 일관성과 유연성


### 데이터 모델링의 순서

1. 개념적 데이터 모델링
- 현업 요구사항을 기반으로 어떤(What) 데이터가 필요한지 결정하는 단계
- 요구사항(줄글)을 ERD(Entity - Relationship Diagram)으로 만든다.
- 비즈니스 이해에 중점. 엔티티, 관계 등 표현. 단순 구조, ERD 등.
2. 논리적 데이터 모델링
- 개념적 모델링한 결과를 토대로 **속성 입력, 식별자 설정, 제약조건, 정규화 등**을 반영하여 보다 상세한 설계도를 만드는 과정.
- 현업의 요구사항을 모두 반영해야 하고 데이터 중복 없게 정규화해야 한다.
- 정확한 설계에 중점. 속성, 식별자, 정규화, 제약조건 등 표현. 요구사항 완전 반영.
3. 물리적 데이터 모델링
- 논리적 모델을 실제 DBMS에 맞게 물리적 속성(**데이터 타입, 크기, 저장 방식 등**)으로 구체화하는 단계
- 실제 구현에 중점. 데이터타입, 크기, 인덱스, 파티션 등 표현. DBMS 구조에 맞춘 구조화.


**순서별 특징**

1. 개념적 데이터 모델링

요구사항은 ERD로 표현하는 과정. 가장 높은 추상화 레벨. 업무 기준으로 포괄적 수준의 모델링 수행.

- 추상화, 단순화, 명확화
- 업무 중심적
- EA 수립에 활용
- DBMS 독립적

*EA : Enterprise Architecture, 전사 아키텍처. 기업 전체의 비즈니스 정보, 인프라, 기술 구조 등을 통합적으로 관리하는 프레임워크.*


2. 논리적 데이터 모델링

위를 바탕으로 데이터 구조를 상세히 설계하는 단계. 실제 구현을 고려해 여러 구체적 요소 반영. 요구사항의 완벽 반영을 위해 반복 설계.

- 모든 요구사항의 반영
- key, 속성, 관계 등 정의
- 재사용성 높음
- DBMS 종속적
- 정규화 수행 *정규화 관해서는 나중에*

그리는 과정

> 속성 입력
>
> 식별자 지정
>
> 정규화
> 
> 관계 설정
>
> N:N 관계 해소


3. 물리적 데이터 모델링

위의 결과를 기반으로, 실제 DB에 적용할 수 있는 구조로 변환하는 과정. SQL을 이용해 테이블을 생성하고, 속성의 자료형, 크기, 저장방식 등 물리적인 요소까지 함께 설계.

- DB가 이해할 수 있는 형태로 변환
- 성능과 저장공간 고려
- 가장 구체적인 모델링 단계
- 반정규화(역정규화)수행

*정규화 : 중복을 줄이고 이상 현상을 막기 위해 엔터티를 분리*

*반정규화 : 성능을 높이기 위해 엔터티를 통합하거나 일부 중복을 허용하는 방식*



### 데이터베이스의 독립성

**데이터 구조가 변경되더라도 다른 계층에 영향을 주지 않는 성질**

독립성은 다른 스키마에 영향을 주지 않으므로 유지보수 및 변경을 유연하게 할 수 있게 된다.

### DB Schema 구조

스키마(Schema) : 데이터베이스를 바라보는 관점(Viewpoint)

**3단계 스키마 구조**
1. 외부 스키마
- 사용자 맞춤 뷰
- 사용자 관점(일반 사용자의 관점에서 바라본 스키마)
- 사용자별로 필요한 데이터만 선택적으로 볼 수 있는 구조
2. 개념(Conceptual) 스키마
- 통합된 논리 구조
- 통합 관점(설계자 혹은 개발자의 관점에서 바라본 스키마)
- 전체 데이터를 통합적으로 표현
3. 내부(Internal) 스키마
- 물리적 관점(실제 물리적 구조를 고려한 스키마)
- 데이터가 실제 저장되는 구조와 방식 등 물리적인 성능 고려

**스키마 구조의 독립성**
1. 논리적 독립성
- 개념 스키마와 외부 스키마 간의 독립성
- 개념 스키마에 변경이 생겨도, 외부 사용자에게 영향을 주지 않는다.
- (예외적으로, 속성 삭제는 영향을 줄 수도 있음.)
2. 물리적 독립성
- 외부, 개념 스키마와 내부 스키마 간의 독립성
- 물리적인 저장 구조(내부 스키마)가 바뀌어도 논리적인 개념 구조(외부, 개념 스키마)에 영향을 주지 않는다.


### 데이터 모델 이해관계자
1. 개발자 : 실제 정보시스템 구축 위해 데이터 모델을 읽을 수 있어야.
2. DBA : 전반적인 DB 관리를 담당하는 전문가.
3. 현업 담당자 : 실제 업무를 이해하는 고객. 개발 과정에 참여해야 함.
4. 전문 모델러 : 일반적으로는 개발자가 겸함. 대형 프로젝트에서는 존재.
