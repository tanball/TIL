# 속성

## 정의

**속성(Attribute)** : 엔터티가 가지는 업무상 관리해야 할 구체적인 정보 항목

엔터티에 속한 인스턴스를 구체적으로 설명하는 요소


## 특징
1. 업무상 필요로 하는 데이터
2. 의미상 더 이상 분리되지 않는 최소의 단위
3. 엔터티의 공통적인 특징을 설명하고, 인스턴스의 구체적인 내용을 설명한다.
4. 주식별자에 함수적 종속해야 한다.
5. 하나의 속성은 하나의 속성값만 가진다

*함수적 종속 : 정규화 개념과 유관. 향후 상세 설명*


## 엔터티 - 인스턴스 - 속성 - 속성값 관계

| 개념 | 정의 | 관계 |
|------|------|------|
| 엔터티 | 저장할 데이터의 집합 | 최소 2개 이상 인스턴스 / 최소 1개 이상 속성 필요 |
| 인스턴스 | 엔터티 안의 개별 데이터 | 속성값들의 조합 |
| 속성 | 인스턴스를 설명하는 항목 | 하나의 속성에 하나의 속성값만 존재해야 함 |
| 속성값 | 속성에 입력되는 실제 값 | 위와 같다. |


## 속성 분류

### 특성별
1. 기본 속성 : 요구사항에서 직접 도출된 속성. 대부분 속성이 속함 (학과명, 학과계열, 학생수 등)
2. 설계 속성 : 설계 편의나 식별을 위해 만든 속성. 업무상 편의 목적, 특정한 규칙 적용 (학과ID)
3. 파생 속성 : 다른 속성들로부터 계산되는 속성. 데이터 편경시마다 재계산해 저장. (학과계열학생수)

*시험 기준으로 파생 속성의 과용을 자제할 것 권고. 시험은 정규화를 보다 중시*

### 엔터티 구성 방식
1. PK(Primary Key) : 논리적 모델링에서의 식별자. 물리적 모델링에서는 primary key로 지칭(의미 동일)
2. FK(Foreign Key) : 외래키. 다른 엔터티와 관계를 맺기 위해 외부에서 빌려온 식별자. 식별자 혹은 UNIQUE KEY만 가능.
3. 일반 속성 : 위 둘 외 나머지. 

*일반 속성 중 다중값인 대상(다중속성)은 정규화 대상*

## 도메인

**도메인(Domain)** : 속성에 입력될 수 있는 데이터의 **타입, 허용 범위, 형식 등**을 정의한 제약 조건의 집합. 속성값의 일관성과 유효성을 유지.


## 속성 명명 규칙
1. 가급적 현업 용어 사용
2. 가급적 불필요한 약어 사용 자제
3. 서술식, 수식어 등 제한. 명사형, 복합명사 사용 권장.
4. 전체 데이터 모델에서 유일한 속성명 확보
