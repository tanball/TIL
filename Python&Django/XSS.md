# XSS 공격

Cross-Site Scripting : 교차 사이트 스크립팅. 공격자가 웹 사이트 / 웹 앱에 웹 스크립트를 통해 악성 코드를 삽입하는 전술. 사용자가 웹 사이트를 로드하거나 웹 앱을 실행하면 이러한 스크립트가 사용자 브라우저에서 실행되게 된다.

![XSS 참조 이미지](https://www.cloudflare.com/img/learning/security/threats/cross-site-scripting/xss-attack.png)

url 끝에 추가하거나, 사용자 생성 콘텐츠를 표시하는 페이지에 직접 삽입하는 방식이 흔하다.

클라이언트 측 코드 삽입 공격의 일종


## 클라이언트 측 코드

사용자의 컴퓨터에서 실행되는 자바스크립트 코드
(서버가 아닌, 사이트에 접속하는 유저=사용자의 컴퓨터에서 실행)

웹브라우저가 웹페이지 로드 -> 클라이언트 측 코드 실행.

장점 : 상호작용마다 서버와 통신할 필요 없음. 상호작용 많은 페이지에서 좋다.

여러 웹에서 사용되며, 유튜부, 페이스북, 트위터 등에도 XSS 공격이 있었음.


## 실제 가능한 방법

1. 반사형 XSS
- 비지속적
- 피싱 이메일, 온라인 게시물 등을 통해 웹사이트 링크로 연결
- 이 때, 링크는 합법적인 실제 사이트 링크
- 단, 링크 뒤에 특정한 코드가 삽입된다(이게 악성 코드)

2. 저장된 XSS
- 지속적
- 콘텐츠(댓글, 포스트, 프로필 등)를 입력하여 개시할 때 악성코드 삽입
- 삽입된 채로 DB에 저장됨 -> 사용자들이 페이지에 액세스할 때마다 실행됨
- 페이지 로드 시 스크립트 실행


## 방지법

### 사용자 입장
1. 의심스러운 이메일/메시지 식별
2. 링크 조사

### 개발자 입장
1. 입력 유효성 검사
- 특정 기준을 충족하지 않고 페이지/양식에 데이터 개시하는 것 차단
- ex) 주민등록번호/전화번호 요청 시 숫자, 대시만 입력 가능.
- XSS 공격에서 자주 쓰이는 태그나 문자를 명시적으로 거부하는 규칙(=유효성 검사 규칙) 설정
- ex) `<script>` 태그 등
- 댓글/게시물/입력폼에 HTML 사용 차단
2. 데이터 삭제
- 데이터를 타 사용자에게 표시하기 전 검사하는 프로세스 구축
- ex) bleach 등으로 출력 전 html 입력 삭제 & 악성코드 필터링
- ex) 출력 인코딩, 이스케이프 처리 등으로 악의적 스크립트나 기타 코드를 일반 텍스트로 변환.
3. 코드 검토 및 테스트
- 웹 앱 개발 전반에 걸쳐 보안 통합.
- 코드 검토, 웨협 모델링과 테스트로 취약점 식별

### 보안팀 입장
1. 콘텐츠 보안 정책(CSP) 구현
2. 보안 쿠키
+자바스크립트가 쿠키에 액세스하지 못하게 차단할 수도.
3. 웹 앱 방화벽 구현
4. 피싱 감지 및 대응에 중점
5. 앱에서 보안 테스트 실행
6. 사고 대응 계획 구축
